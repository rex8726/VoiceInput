# Mac 语音输入软件 PRD

## 1. 产品概述

本产品是一款面向 macOS 的纯粹语音输入工具。用户通过全局快捷键开始和结束讲话，系统将语音转写为文本，并通过 AI 后处理自动去除口水词、补全标点、整理段落和顺序，最终把处理后的文本自动输入到当前光标所在位置，或复制到剪贴板供用户手动粘贴。

产品不做语音助手、不做翻译、不做知识问答，也不主动替用户扩写观点。核心目标是让用户可以像自然讲话一样输入文字，同时得到一段忠实、清晰、可直接使用的文本。

## 2. 背景与问题

用户在 Mac 上长时间打字时，常见痛点包括：

- 想法产生速度快于打字速度。
- 中文输入、长文本输入、Prompt 输入、聊天回复都容易被输入法和键盘节奏打断。
- macOS 自带听写更偏“原样转写”，对口语里的重复、犹豫、改口、无意义停顿处理不够。
- 通用 AI 聊天工具可以润色文本，但需要复制、粘贴、发给模型、再复制回来，流程太重。

本产品要解决的是：用户在任意输入框中自然说话，系统自动把口语整理成符合原意的书面文本，并快速回填到当前位置。

## 3. 产品目标

### 3.1 核心目标

- 用户可以通过一个简单快捷键完成语音输入。
- 系统可以将语音准确转写为文本。
- 系统可以对转写结果进行 AI 后处理，去除口水词、修正明显语序问题、添加标点和分段。
- 处理后的文本必须忠实保留用户原始本意。
- 处理结果可以自动进入当前输入框，或至少复制到剪贴板。

### 3.2 非目标

当前阶段不做以下能力：

- 翻译。
- 问答。
- 总结网页或文档。
- 复杂命令执行。
- 跨设备同步。
- 团队协作。
- 会议转录。
- 长音频文件批量转写。
- 多角色对话记录。

## 4. 目标用户

### 4.1 主要用户

高频使用 Mac 输入中文或中英混合文本的人，包括：

- 创作者。
- 产品经理。
- 开发者。
- 研究人员。
- 创业者。
- 需要频繁写 Prompt、写消息、写文档的人。

### 4.2 典型使用场景

- 在 ChatGPT、Claude、Codex、Cursor 等 AI 工具中输入长 Prompt。
- 在微信、飞书、Slack、邮件中回复消息。
- 在 Notion、Obsidian、备忘录中记录想法。
- 在浏览器文本框中填写内容。
- 在 IDE 或终端辅助工具中输入自然语言描述。

## 5. 核心用户故事

### 5.1 快速语音输入

作为用户，我希望把光标放在任意输入框中，按下全局快捷键开始讲话，再按同一快捷键结束讲话，系统自动把整理后的文字输入到当前位置。默认快捷键为 Option + 1；Fn 可在设置中启用，但默认关闭以避免和 macOS 输入法切换冲突。

验收标准：

- 用户可以用全局快捷键触发录音。
- 录音期间有清晰但不打扰的视觉状态提示。
- 停止录音后，系统自动处理语音。
- 处理完成后，文本进入当前输入框或复制到剪贴板。

### 5.2 口语整理

作为用户，我希望自己讲话时可以自然停顿、重复、改口，系统最终输出一段顺畅文字。

验收标准：

- 自动去除明显口水词，例如“呃”“嗯”“然后就是”“那个”等。
- 自动处理重复表达。
- 自动识别轻微改口，保留用户最终表达。
- 自动添加中文标点。
- 根据内容自然分段。
- 不改变用户观点，不擅自增加新事实。

### 5.3 结果可控

作为用户，我希望在自动输入失败时，仍然可以通过剪贴板拿到结果。

验收标准：

- 自动粘贴失败时，处理结果仍保留在剪贴板。
- 用户可以从菜单栏最近一次结果中复制。
- 失败状态需要有明确但克制的提示。

## 6. 功能范围

### 6.1 录音触发

第一版支持：

- 默认全局快捷键：Option + 1。
- 按一次快捷键开始录音，再按一次结束录音。
- 录音中再次按快捷键即停止并进入处理流程。
- 可选启用 Fn 键触发，但默认关闭，避免和输入法切换冲突。

后续可选：

- 按住 Fn 录音，松开结束。
- 自定义快捷键。
- 取消本次录音快捷键，例如 Esc。

### 6.2 视觉交互小组件

录音和处理期间显示一个小型悬浮组件。

组件状态：

- Idle：默认不显示，或只在菜单栏显示。
- Recording：显示正在录音状态。
- Processing：显示正在转写/整理状态。
- Done：短暂显示完成状态。
- Error：短暂显示错误状态，并说明是否已复制原始结果。

组件设计原则：

- 小而轻，不遮挡主要内容。
- 不做复杂窗口。
- 状态一眼可识别。
- 不显示大段说明文字。
- 支持深色模式和浅色模式。

建议组件元素：

- 圆形或胶囊型悬浮条。
- 麦克风图标。
- 动态声波或音量脉冲。
- 简短状态文字，例如“正在听”“整理中”“已输入”。
- 录音时长。

### 6.3 语音转写

输入：

- 用户录制的音频。

输出：

- 原始转写文本。

要求：

- 优先支持中文。
- 支持中英混合。
- 对短句和中长段落都要稳定。
- 需要有超时和错误处理。

### 6.4 AI 后处理

输入：

- 原始转写文本。
- 可选上下文：当前 App 名称、用户选择的处理模式。

输出：

- 整理后的最终文本。

处理原则：

- 忠实保留用户原始本意。
- 不扩写。
- 不补充用户没有说过的信息。
- 不改变事实、数字、人名、项目名。
- 修正明显语音识别错误时要谨慎。
- 允许把口语变成自然书面语。

第一版处理能力：

- 添加标点。
- 自然分段。
- 去口水词。
- 去无意义重复。
- 整理轻微语序。
- 保留原始语气。

后续可选模式：

- 原样转写。
- 轻度整理。
- 标准整理。
- 更正式。
- 更简短。

第一版默认使用“标准整理”。

### 6.5 文本输入

第一版支持两种输出方式：

- 自动粘贴到当前光标位置。
- 同时复制到剪贴板，作为兜底。

输入策略：

- 默认自动粘贴，并同时复制到剪贴板。
- 优先使用剪贴板加模拟 Cmd+V。
- 如果自动粘贴失败，保留剪贴板结果并提示用户手动粘贴。
- 如果当前没有输入框或目标 App 不接受粘贴，结果保留在剪贴板，用户自行粘贴。

权限要求：

- 麦克风权限。
- 辅助功能权限，用于全局快捷键监听或自动粘贴。
- 可能需要输入监控权限，取决于 Fn 键监听实现。

### 6.6 菜单栏入口

第一版提供菜单栏图标。

菜单项建议：

- 开始/停止录音。
- 复制最近一次结果。
- 打开设置。
- 授权状态。
- 退出。

### 6.7 设置

第一版设置项：

- API Base URL。
- API Key。
- 语音转写模型。
- 文本整理模型。
- 自动粘贴开关。
- 处理完成后保留到剪贴板开关。
- 默认处理模式。
- 历史记录数量：默认最近 10 条。

安全要求：

- API Key 不写入仓库。
- API Key 存储在 macOS Keychain，或本地用户配置中并明确提示风险。
- 日志不输出 API Key。
- 错误信息不包含完整 API Key。

## 7. 交互流程

### 7.1 主流程

1. 用户把光标放在目标输入框。
2. 用户按 Fn。
3. 悬浮组件出现，进入 Recording 状态。
4. 用户讲话。
5. 用户再次按 Fn。
6. 系统停止录音，进入 Processing 状态。
7. 系统调用语音转写模型，得到原始文本。
8. 系统调用文本整理模型，得到最终文本。
9. 系统把最终文本复制到剪贴板。
10. 若开启自动粘贴，则模拟 Cmd+V 输入到当前光标位置。
11. 悬浮组件短暂显示 Done 状态后隐藏。

### 7.2 异常流程

无麦克风权限：

- 显示错误状态。
- 引导用户打开系统设置授权。

无辅助功能权限：

- 可以录音和处理。
- 无法自动粘贴。
- 结果复制到剪贴板。
- 提示用户手动粘贴或开启权限。

转写失败：

- 显示错误状态。
- 不粘贴。
- 保留音频临时文件用于本次重试，或直接丢弃。

AI 后处理失败：

- 可选择粘贴原始转写文本。
- 默认建议：复制原始转写到剪贴板，并提示“整理失败，已复制原始转写”。

网络失败：

- 显示错误状态。
- 不丢失当前录音处理状态。
- 支持重试最近一次录音。

## 8. 模型与 API 设计

用户提供硅基流动 API，平台内有多个模型可调用。产品需要把模型调用做成可替换配置。

### 8.1 推荐模型链路

第一阶段：

- STT 模型：负责音频转文字。
- LLM 模型：负责文本整理。

不建议第一版直接用单一多模态模型完成全流程，除非目标 API 的稳定性和速度更好。

### 8.2 抽象接口

配置字段：

- `base_url`，默认 `https://api.siliconflow.cn/v1`
- `api_key`
- `stt_model`，默认候选 `FunAudioLLM/SenseVoiceSmall`
- `text_model`
- `timeout_seconds`
- `language`

### 8.3 硅基流动接口形态

语音转文字：

- Endpoint：`POST /v1/audio/transcriptions`
- Content-Type：`multipart/form-data`
- 鉴权：`Authorization: Bearer <API_KEY>`
- 参数：`file`、`model`
- 返回：`{ "text": "<string>" }`
- 文档列出的 STT 模型：`FunAudioLLM/SenseVoiceSmall`、`TeleAI/TeleSpeechASR`
- 文档限制：音频时长不超过 1 小时，文件大小不超过 50MB。

文本整理：

- Endpoint：`POST /v1/chat/completions`
- Content-Type：`application/json`
- 鉴权：`Authorization: Bearer <API_KEY>`
- 参数：`model`、`messages`、`temperature`、`max_tokens` 等。
- 第一版建议关闭流式输出，降低实现复杂度。
- 对支持 thinking 的模型，默认关闭 `enable_thinking`，优先降低延迟并避免输出解释内容。

### 8.4 文本整理 Prompt 原则

系统提示词方向：

```text
你是一个语音输入文本整理器。你的任务是把语音转写文本整理成清晰、自然、可直接发送的中文文本。

要求：
1. 忠实保留用户原意。
2. 不添加用户没有表达的新信息。
3. 去除口水词、停顿词和无意义重复。
4. 自动添加标点。
5. 根据语义自然分段。
6. 轻微调整语序，使文本更顺畅。
7. 保留数字、人名、产品名、代码名和专有名词。
8. 只输出整理后的文本，不解释。
```

### 8.5 模型选择标准

STT 模型优先级：

- 中文识别准确率。
- 中英混合识别能力。
- 延迟。
- 短音频稳定性。
- 长音频稳定性。
- 成本。

文本模型优先级：

- 忠实度。
- 中文润色自然度。
- 不擅自扩写的稳定性。
- 延迟。
- 成本。

## 9. 数据与隐私

第一版默认策略：

- 音频只用于本次转写。
- 原始音频处理完成后删除。
- 本地保存最近 10 条原始转写和最终结果，便于用户回看和复制。
- 不保存原始音频。
- 不记录 API Key。
- 日志中不包含完整用户语音内容，除非用户开启调试模式。

后续可选：

- 本地历史开关。
- 本地词典。
- 完全本地 STT。
- 敏感 App 黑名单。

## 10. 权限需求

macOS 权限：

- Microphone：录音必需。
- Accessibility：自动粘贴和全局快捷键可能需要。
- Input Monitoring：监听 Fn 可能需要。

需要在首次启动时清晰展示权限状态，但避免把用户带进复杂说明页。

## 11. 成功指标

第一版主观指标：

- 用户能在 3 秒内理解怎么开始录音。
- 用户说一段 10 到 30 秒的话，输出文本基本可直接使用。
- 用户无需手动复制粘贴，或只在权限不足时手动粘贴。
- 输出文本没有明显改变原意。

第一版可量化指标：

- 10 秒语音从停止录音到结果输入，目标小于 5 秒。
- 自动粘贴成功率大于 95%。
- 常见中文口水词清理成功率大于 90%。
- AI 后处理导致原意偏移的情况低于 5%。

## 12. MVP 范围

MVP 必做：

- macOS 菜单栏 App。
- Fn 切换录音开始/结束。
- 悬浮录音状态组件。
- 音频录制。
- 调用 STT API。
- 调用 LLM 做文本整理。
- 自动复制到剪贴板。
- 可选自动粘贴。
- 基础设置页。
- 权限状态提示。

MVP 暂不做：

- 翻译。
- AI 问答。
- 长历史管理。
- 文件转写。
- iOS/Windows。
- 团队版。
- 云端账号系统。

## 13. 视觉小组件需求

### 13.1 位置

默认显示在屏幕底部中间，或当前鼠标附近。第一版建议底部中间，稳定且不依赖复杂定位。

### 13.2 尺寸

建议尺寸：

- 宽度：180 到 260 px。
- 高度：44 到 56 px。
- 圆角：胶囊形。

### 13.3 状态设计

Recording：

- 麦克风图标。
- 红色或系统强调色小点。
- 动态声波。
- 录音时长。
- 文案：“正在听”。

Processing：

- 旋转进度图标。
- 文案：“整理中”。

Done：

- 对勾图标。
- 文案：“已输入”或“已复制”。

Error：

- 警示图标。
- 文案：“处理失败”或“已复制原文”。

### 13.4 交互规则

- 录音开始时淡入。
- 处理完成后 1 到 1.5 秒淡出。
- 不抢焦点。
- 不影响当前输入框。
- 不需要用户点击。

## 14. 技术初步方案

建议技术栈：

- Swift / SwiftUI。
- AVFoundation 录音。
- Carbon 或 CGEventTap 监听全局按键。
- NSPasteboard 写入剪贴板。
- CGEvent 模拟 Cmd+V。
- URLSession 调用 API。
- Keychain 保存 API Key。

主要模块：

- `HotkeyManager`：监听 Fn。
- `AudioRecorder`：录音与音频文件管理。
- `TranscriptionClient`：调用 STT。
- `TextRefinerClient`：调用 LLM 后处理。
- `PasteboardService`：复制和自动粘贴。
- `OverlayWindow`：悬浮状态组件。
- `SettingsStore`：设置。
- `KeychainStore`：密钥。

## 16. 待确认问题

- Fn 键交互是“按一次开始，再按一次结束”，还是“按住开始，松开结束”？当前 PRD 默认前者。
- 是否允许用其他快捷键作为兜底，例如 Option+Space？
- API 是否兼容 OpenAI 格式？
- API 是否提供音频转写模型，还是只有文本/多模态模型？
- 希望默认自动粘贴，还是默认只复制到剪贴板？
- 是否需要本地保存最近 N 条历史？
- 视觉组件偏向极简系统风，还是更接近 Typeless 的强反馈风格？
